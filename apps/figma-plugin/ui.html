The 3-layer model

A healthy design system separates raw values from meaning from usage. Each layer references the one below it — never the other way around.

1. Primitives
The full palette. Raw color, spacing, and type values. Named by what they are, not what they do.
2. Themes
Semantic aliases that point to primitives. Named by intent — they enable light/dark mode and brand theming.
3. Components
Component-scoped tokens that reference theme tokens. Tightly bound to a specific UI element.

Color naming

Good
primitives/blue-500 themes/color/bg/primary components/button/bg/default
Bad
blue (no layer, ambiguous scope) primary-bg (flat — mixes intent with layer) button-color (no hierarchy, unclear property)

Spacing & sizing

Good
primitives/spacing/400 themes/spacing/element/gap components/card/padding
Bad
16px (raw value, no token) spacing-medium (flat, not layered) card-gap (no clear layer or hierarchy)

Why it matters

Without layers — theme changes require touching hundreds of components. Renaming a color breaks everything. Dark mode is manual.

With layers — swap a theme and every component updates. Primitives are stable. Components are self-contained and portable.
